# AI 小说拆解工作台统一产品与系统设计基线 V1.0

- 更新时间：2026-07-18
- 文档性质：当前唯一产品与系统判断入口
- 主要读者：产品设计者、开发者、后续接手的 AI
- 权威规则：本文件管理当前目标、范围、页面、流程、参考机制、进度和开发顺序；单项目研究档案只提供证据，不直接决定产品功能。

## 1. 为什么重新统一

旧资料分别描述了产品定义、候选架构、研究结论、工程可靠性、路线图和当前进度。它们在各自阶段有价值，但已经出现以下问题：

- 旧产品文档以 30～50 章为首轮规模，现在产品已经明确支持短篇到百万字长篇。
- 旧文档中的 M0～M8 与当前 P0～P6 使用了不同阶段含义。
- “研究过某个项目”有时被误解成“要照搬那个项目的功能”。
- 产品目标、当前进度、下一步和参考来源散落在多份文件中。
- 临时实现中的参数和界面决定没有先回查产品需求与参考机制。

从 V1.0 开始，本文件是唯一活动基线。旧产品、架构、研究汇总和 M0 方案保留在历史归档中，只用于追溯当时为什么这样判断。

## 2. 产品定义

### 2.1 产品目标

本项目要交付一个可长期浏览、核对和操作的中文小说拆解工作台，不是一次性报告生成器，也不是小说续写工具。

用户一次导入整本小说，系统根据篇幅自动分层处理。首次拆解按阶段引导，用户查看阶段总览、抽查原文并确认后进入下一阶段。完成后，用户可以在人物、剧情、事件、时间线、事实与状态、世界设定、伏笔、冲突和节奏等版块中持续浏览，并从重要结论跳回来源章节和原文上下文。

导出报告是附加能力，不是产品主体。

### 2.2 核心用户

- 小说创作者和编剧：复盘长篇结构、人物变化、伏笔、冲突和世界规则。
- 编辑和内容策划：检查章节功能、节奏、逻辑矛盾和承诺回收。
- 拆书研究者和深度读者：建立可回查的事件时间线、人物关系和跨章分析。

### 2.3 首版边界

- 中文小说优先，单用户、本机保存项目数据。
- 只设计、测试和验收 PC/桌面端浏览器，不做手机端。
- 支持短篇、中长篇、长篇和百万字以上作品，不设置统一小说字数硬上限。
- 第一批导入格式：TXT、Markdown、DOCX、EPUB。
- 首版使用用户配置的在线大语言模型分析，不要求用户运行本地模型。
- 质量优先，整本分析可以持续数小时，但必须显示阶段、进度、失败原因和下一步。
- 第一版先完成核心拆解；主题、文风、写作技巧和正式报告导出后置。

### 2.4 明确不做

- 不做小说续写、仿写或整书改写。
- 不做角色扮演聊天产品。
- 不因为参考项目存在“世界书”“角色卡”或“聊天记忆”就照搬同名功能。
- 不做多人协作、团队权限、云端账号和商业计费。
- 不把内部任务编号、租约代次、数据库编号等开发信息展示给普通用户。
- 不把模型生成的流畅文字直接当作可靠事实。

## 3. 设计原则

### 3.1 需求决定零件，不由参考项目决定功能

参考项目是机制、思路和实现零件的来源，不是产品功能清单。研究一个功能时必须区分四层：

1. 原项目功能，例如世界书、角色卡、聊天记忆。
2. 底层机制，例如条件触发、检索排序、上下文预算、版本更新。
3. 我们的真实需求，例如分析某章时找到相关人物、设定和历史证据。
4. 我们的最终功能，例如证据检索、世界设定回查或人物状态查询。

只有真实需求成立、底层机制合适时，才重新设计成我们的功能。不得继承原项目的名称、边界和使用方式。

### 3.2 证据优先

- 原文是最高权威来源。
- 每个重要人物、事件、事实和分析结论必须能回到原文章节与上下文。
- 摘要、图谱、社区报告和自然语言分析都是派生结果，不能冒充原文。
- 支持证据、反面证据和不确定性必须显式保存。

### 3.3 结构优先

先建立可校验的人物、事件、事实、状态和分析主张，再生成自然语言解释。模型负责提出语义候选；程序负责稳定编号、原文位置、校验、版本、状态和写入。

### 3.4 质量上限优先，但必须实测

首版先证明质量上限，不为了省调用提前删除可能有价值的机制；同时也不因为参考项目存在某项机制就全部启用。每条高成本路线都必须通过样本、金标和对照实验确认收益。

### 3.5 可恢复、可诊断

长流程必须可以暂停、继续、重试和局部重算。失败时必须说明失败阶段、原因、影响范围和下一步，不能只显示“分析失败”。

## 4. 用户使用流程

```text
创建小说项目
  ↓
导入整本 TXT / Markdown / DOCX / EPUB
  ↓
检查卷章、正文、顺序、空章、重复章和编码问题
  ├─ 有问题：定位原文并修正或重新导入
  └─ 无问题：确认章节结构
  ↓
人物识别与归一
  ├─ 查看总览、别名和证据
  └─ 确认或标记问题
  ↓
剧情阶段与事件识别
  ├─ 查看跨章事件、参与人物和证据
  └─ 确认或标记问题
  ↓
事实、世界设定、人物状态和时间线
  ↓
伏笔、冲突、节奏和场景功能
  ↓
进入统一工作台持续浏览、搜索、关联跳转和回查原文
```

每个阶段默认停在确认页。用户确认后才进入下一阶段；标记问题时系统优先判断问题来自原文解析、程序规则、提示词还是模型能力，不把人工修正当作掩盖系统问题的默认方案。

## 5. 页面与导航

首版采用 PC 端左侧主导航，不使用移动端底部导航。

| 页面 | 用户目标 | 主要内容 | 关键异常状态 |
|---|---|---|---|
| 项目首页 | 创建、进入和继续小说项目 | 项目列表、最近进度、继续分析 | 文件丢失、项目损坏、上次任务中断 |
| 导入与章节确认 | 得到可信的整本章节结构 | 文件导入、卷章树、正文预览、问题提示 | 编码错误、标题误识别、空章、重复章 |
| 阶段确认页 | 抽查当前阶段结果并决定是否继续 | 总览、原文证据、问题标记、确认按钮 | 结果为空、部分批次失败、证据无法定位 |
| 拆解工作台 | 浏览整本小说的各类拆解结果 | 总览、人物、剧情、事件、时间线、事实状态、世界设定、伏笔、冲突、节奏 | 数据未生成、结果过期、受上游修订影响 |
| 原文阅读与证据查看 | 理解结论来自哪里 | 章节树、原文、证据高亮、上下文 | 原文版本变化、证据位置失效 |
| 问题中心 | 集中处理系统发现和用户标记的问题 | 问题类型、影响对象、建议操作、重算范围 | 问题互相依赖、修改影响下游结果 |
| 运行记录 | 了解长任务当前做到哪里 | 阶段进度、耗时、费用、失败原因、继续/停止 | 网络中断、限流、模型输出错误 |
| 设置中心 | 配置在线模型和分析方案 | 模型服务、模型测试、任务方案、数据与隐私 | 密钥错误、模型无权限、参数不支持 |

工作台内部导航固定为：总览、原文、人物、剧情、事件、时间线、事实与状态、世界设定、伏笔、冲突、节奏、问题。未完成的版块明确显示“尚未分析”或“本版本尚未开放”，不能用模拟内容冒充结果。

## 6. 核心功能范围

| 阶段 | 用户可见目标 | 必须完成的能力 | 当前状态 |
|---|---|---|---|
| P0 工程地基 | 长任务可靠运行 | 任务领取、恢复、重试、取消、制品幂等、自动测试 | 基本完成 |
| P1 整本导入 | 正确读取并确认整本小说 | 四种格式、卷章正文、顺序、编码、空章、重复章、原文坐标 | 首条闭环，尚未验收 |
| P2 人物、剧情与事件 | 得到真实人物和跨章事件并回查原文 | 人物候选、别名归一、剧情阶段、事件边界、跨章合并 | 只有人物事件候选首条闭环 |
| P3 事实与状态 | 查询某一章节时成立的事实 | 世界规则、事实版本、人物状态、角色认知、时间查询 | 未开始 |
| P4 核心分析 | 浏览可核验的伏笔、冲突和节奏 | 分析主张、支持/反面证据、伏笔回收、冲突升级、节奏和场景功能 | 未开始 |
| P5 统一工作台 | 在各版块间搜索、跳转和处理问题 | 正式导航、关联跳转、阶段确认、问题中心、局部重算 | 未开始 |
| P6 长篇验收 | 整本长篇连续完成 | 5 万、50 万、100 万字以上样本，质量、时间、费用、恢复和可读性 | 未开始 |

## 7. 信息与分析分层

```text
原始小说与版本
  ↓
章节、段落和原文证据位置
  ↓
人物 / 事件 / 事实候选
  ↓
人物归一、事件合并、事实版本与状态
  ↓
剧情阶段、时间线、关系和世界设定视图
  ↓
伏笔、冲突、节奏等带证据的分析主张
  ↓
工作台页面与导出结果
```

上游修改必须明确标记受影响的下游结果。原文、正式对象、派生视图和自然语言报告必须分开保存。

## 8. 在线模型与分析方案

### 8.1 设置结构

- 模型服务：服务名称、类型、接口地址、访问密钥、连接状态和模型目录。
- 分析方案：某类任务使用哪个服务、哪个模型、哪些参数和什么失败策略。
- 一个服务可以服务多个任务；人物抽取、事件分析和全书归纳不应被迫共用相同参数。

### 8.2 连接成功的层级

界面必须区分：

1. 配置已经保存。
2. 地址和密钥可以访问。
3. 模型目录读取成功。
4. 所选模型拥有调用权限。
5. 所选模型支持当前任务需要的结构化输出和参数。

“读取到 20 个模型”不能显示成“该模型已经可用于拆书”。

### 8.3 参数原则

- 默认使用“自动”，即在没有任务实测依据时不强行发送温度、采样和推理参数。
- 模型或服务不支持的参数不显示、不发送。
- 参数范围、步长和默认值必须来自服务说明、模型能力或质量实验，不得凭界面整齐程度自行设定。
- 温度不是全局质量旋钮。结构化抽取、主观文学分析和总结任务分别通过实验确定方案。
- 每次运行记录实际服务、模型、提示词版本和实际发送参数。

### 8.4 失败处理

至少区分地址错误、密钥无效、模型无权限、参数不支持、结构化输出不支持、限流、超时、服务故障和输出格式错误，并给出通俗的处理建议。

## 9. 参考项目零件吸收规则

### 9.1 决策类型

| 类型 | 含义 |
|---|---|
| 直接吸收 | 需求、机制和边界一致；按许可证要求采用或独立实现 |
| 改造吸收 | 机制有用，但输入、输出、命名或使用流程需要重做 |
| 只学思想 | 只保留设计思想，不继承代码和原产品结构 |
| 明确舍弃 | 与产品目标冲突、证据不足或成本大于收益 |
| 必须自研 | 参考项目没有满足需求的现成机制 |

### 9.2 当前零件矩阵

| 我们需要的零件 | 参考证据 | 吸收内容 | 不照搬 | 当前判断 |
|---|---|---|---|---|
| 小说导入与任务组织 | P01 | 导入流程、长任务组织、停止与续跑思路 | 粗固定切块、把局部摘要当全书事实 | 改造吸收，章节解析需自研 |
| 模型连接与方案 | P19、P01 | 服务/模型/预设分层、模型目录、手工回退、能力过滤 | 聊天专用参数和界面、统一固定温度 | 改造吸收 |
| 长篇证据与来源链 | P05、P07、P15 | 稳定来源、逐层回源、分析主张证据 | 摘要替代原文、自由文本位置 | 直接吸收机制，独立实现 |
| 人物候选与归一 | P06、P13 | 候选发现、别名、冲突和可逆合并 | 名字直接作唯一身份、无保护自动合并 | 改造吸收 |
| 事件发现与跨章合并 | P14、P18 | 多路线候选、事件提及、核心事件、时间与关系 | 最近同词定位、强制单词边界 | 改造吸收，边界解析需自研 |
| 事实版本与指定时点状态 | P13、P17 | 有效区间、冲突、状态重放 | 当前值覆盖历史、快照冒充事实 | 直接吸收机制，独立实现 |
| 角色知道什么 | P12 | 客观事实与角色认知分离 | 把全局事实直接暴露给角色视角 | 改造吸收 |
| 上下文选择与检索 | P07、P08、P11、P12、P16 | 任务路由、多路召回、重排、来源回填、预算 | 多套检索链机械叠加 | 按任务改造吸收 |
| 世界设定相关信息触发 | P03、P06、P09 | 条件触发、相关信息选择、版本和状态机制 | 不做“世界书”产品，不继承其字段和编辑流程 | 只吸收机制，转化为设定/证据检索 |
| 伏笔、冲突和专项分析 | P10、P15、P17 | 分析主张、承诺账本、支持与反面证据、派生制品隔离 | 先写长报告再补证据 | 改造吸收，部分需自研 |
| 全局结构与图视图 | P16 | 层级社区、全局/局部探索思路 | 社区报告作为事实证据 | 延后验证 |
| 任务恢复与制品 | P08、P09、P12、P17 | 程序控制、幂等写入、阶段制品和恢复链 | 模型直接改变权威状态 | 已作为 P0 地基实现 |

### 9.3 回查纪律

- 开发某个零件前，先在本矩阵中确认用户需求和参考项目。
- 再回查单项目档案记录的固定仓库、版本、文件和函数位置。
- 参考源码只能在许可证允许且完成登记后引入；默认独立实现机制。
- 找不到合适零件时才自研，不用临时猜测冒充成熟设计。
- 新研究必须对应一个明确缺口；不再为了“多研究一个项目”增加档案。

## 11. 当前开发顺序

1. 完成本文档收口，停止使用互相冲突的旧活动文档。
2. 按零件矩阵补查 SillyTavern 等项目中与当前需求相关、尚未核对的机制。
3. 修正模型设置语义：自动参数、真实模型测试和能力过滤。
4. 完成 P1 复杂导入样本和章节校正体验。
5. 完成 P2 人物归一、剧情阶段和跨章事件。
6. 第一次正式用户验收：整本导入、真实人物事件和原文回查。
7. 按 P3～P5 逐步建设事实状态、核心分析和统一工作台。
8. 使用 5 万、50 万和 100 万字以上样本完成 P6 验收。

## 12. 用户验收边界

用户只验收产品功能和使用效果，不验收分支、PR、数据库迁移、任务租约和内部编号。

正式验收节点：

1. 能导入真实整本小说、正确显示章节，并得到真实人物和事件结果。
2. 人物、剧情、事件、事实、伏笔、冲突等核心版块可以完整浏览和回查原文。
3. 整本长篇能够从导入连续运行到完整工作台。

## 13. 文档与版本规则

- 本文件是唯一当前设计入口；版本从“统一基线 V1.0”重新开始。
