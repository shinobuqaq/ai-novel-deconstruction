# AI 小说拆解工作台统一产品与系统设计基线 V1.0

- 更新时间：2026-07-20
- 文档性质：当前唯一产品、系统、数据、质量和开发判断入口
- 主要读者：产品设计者、开发者、测试者和后续接手的 AI
- 使用规则：后续开发应能只读本文件理解产品目标、用户流程、核心对象、系统边界、质量标准、当前进度和下一步；不再依赖已删除的旧总览、旧路线图或旧架构文档补全背景。

## 1. 文档角色和唯一入口规则

### 1.1 为什么需要统一基线

项目曾分别维护产品定义、候选架构、研究汇总、工程可靠性、路线图、实现审计和问题台账。这些文档在当时有价值，但后来出现了规模、阶段编号、产品边界和当前进度互相冲突的问题。

本文件不是旧文档的摘要，而是它们仍有效内容的完整继承者：

- 产品目标、用户、输入、输出、页面、流程和验收统一在这里。
- 系统形态、数据所有权、核心对象、状态、运行链和失败恢复统一在这里。
- P01～P19 的有效机制、负面边界和实施回查入口统一在这里。
- 当前实现、未完成事项、开发顺序和用户验收节点统一在这里。
- 已过时或已完成的阶段性计划不再保留为活动文档。

### 1.2 文件职责

| 文件类型 | 职责 | 是否决定当前产品 |
|---|---|---|
| 本文件 | 完整产品与系统设计、当前进度、开发顺序和验收 | 是，唯一入口 |
| P01～P19 单项目档案 | 固定版本源码证据、正反例和精确回查位置 | 否，只提供证据和零件 |
| `adr/` | 已接受且需要长期解释的架构决策 | 不能改变本文件目标 |

| `THIRD_PARTY_CODE.md` | 外部源码、许可证与通知登记 | 只管理合规 |
| 代码、迁移和自动测试 | 当前实现与可执行验证 | 实现事实的最高证据 |

### 1.3 变更规则

- 产品目标、用户流程、页面、权威对象或质量门槛改变时，先更新本文件，再改代码。
- 参数、阈值和默认值必须有服务说明、模型能力、质量实验或明确保护规则；没有依据时使用“自动”或标记为实验设置。
- 改变数据所有权、稳定身份、事实与状态分离、主张优先或依赖传播时，必须增加 ADR（架构决策记录）并说明迁移影响。
- 替换模型服务、向量后端、前端库或任务执行器，只要接口和行为契约不变，不需要改写核心产品定义。
- 旧结论中仍有效的信息必须先迁入本文件；迁移确认后删除旧总览，不保留多个需要拼接阅读的入口。

## 2. 产品定义

### 2.1 产品目标

本项目要交付一个可长期浏览、核对和操作的中文小说拆解工作台，不是一次性报告生成器，也不是小说续写工具。

用户一次导入整本小说，系统根据篇幅自动分层处理。首次拆解按阶段引导，用户查看阶段总览、抽查原文并确认后进入下一阶段。完成后，用户可以持续浏览人物、剧情、事件、时间线、事实与状态、世界设定、伏笔、冲突和节奏，并从重要结论回到来源章节和原文上下文。

产品真正要回答的不是“这本书大概讲了什么”，而是：

- 原文中有哪些可验证的人物、事件、事实和状态变化。
- 同一对象如何跨章节演变、合并、冲突或失效。
- 剧情阶段、人物弧、伏笔、冲突和节奏如何形成。
- 每项分析有哪些支持证据、反面证据和不确定性。
- 原文或设置改变后，哪些结果需要重新分析。

导出报告是附加能力，不是产品主体。

### 2.2 产品北极星

对于任意篇幅的中文小说，系统应在不把摘要当事实、不让模型自行创造稳定身份、不把检索结果冒充原文证据的前提下，重建“谁在何时做了什么、事实如何变化、角色知道什么、叙事如何推进”，并让每项重要分析都能回到具体原文。

### 2.3 核心用户和价值

| 用户 | 主要需求 | 产品价值 |
|---|---|---|
| 小说创作者、编剧 | 复盘长篇结构、人物变化、伏笔、世界规则和前后矛盾 | 比手工翻章更快定位证据和状态变化 |
| 编辑、内容策划 | 检查章节功能、节奏、冲突升级、承诺回收和结构缺口 | 获得可核验的分析，而不是只有“感觉” |
| 拆书研究者、深度读者 | 建立事件时间线、人物关系和跨章机制分析 | 可以追溯每个结论的来源 |

首版不为只想得到几百字剧情简介、多用户协作、团队权限、商业计费或公开传播原文的场景设计。

### 2.4 核心用户任务

| 用户任务 | 系统应完成 | 结果 |
|---|---|---|
| 导入长篇小说 | 建立章节、原文位置和稳定来源索引 | 后续结果都能回查 |
| 看到人物多个名字 | 归一身份并保留别名、置信度和争议 | 不拆散人物弧，也不误合并 |
| 追踪跨章事件和事实 | 建立事件时间线、事实版本和指定章节状态 | 能查询当时真实状态 |
| 阅读拆解结论 | 显示支持、反证、状态和来源 | 用户能判断可信度 |
| 修订或替换原文 | 标记受影响对象并局部重算 | 不让旧结果污染新版本 |
| 比较分析路线 | 显示质量、令牌、调用、时间和成本 | 决定哪些机制值得保留 |

### 2.5 首版边界

- 中文小说优先，单用户，本机保存项目数据。
- 只设计、测试和验收 PC/桌面端浏览器，不做手机端。
- 支持短篇、中长篇、长篇和百万字以上作品，不设置统一小说字数硬上限。
- 第一批导入格式：TXT、Markdown、DOCX、EPUB。
- 首版使用用户配置的在线大语言模型，不要求用户运行本地模型。
- 质量优先，整本分析可持续数小时，但必须显示阶段、进度、失败原因和下一步。
- 第一版先完成核心拆解；主题、文风、写作技巧、复杂全局探索和正式报告导出后置。

### 2.6 明确不做

- 不做自动续写、仿写、整书改写或作者文风复制。
- 不做角色扮演聊天产品。
- 不因为参考项目存在“世界书”“角色卡”或“聊天记忆”就照搬同名功能。
- 不做多人协作、团队权限、云端账号和商业计费。
- 不把内部任务编号、租约代次、数据库编号和制品编号展示给普通用户。
- 不把自然语言摘要、社区报告、向量结果或流畅模型回答直接当作事实。
- 不让模型生成或选择任意数据库 ID，不让模型直接执行 SQL、代码或状态写入。

## 3. 产品原则

### 3.1 原文与证据优先

- 原文是最高权威来源。
- 每个正式人物、事件、事实和分析主张必须引用一个或多个原文证据。
- 摘要、图谱、社区报告、状态视图和自然语言解释都是派生结果，不能覆盖原文。
- 支持证据、反面证据、争议和未知必须显式保存。

### 3.2 结构先于报告

先产生可校验的人物、事件、事实、状态和分析主张，再编译自然语言说明。报告只能读取已经验证的对象，不能反向把编辑后的文字写成事实。

### 3.3 模型提议，程序决定

模型适合提出语义候选、判断相似性和生成解释；程序必须拥有：

- 稳定身份和来源位置。
- 数据结构和必填字段校验。
- 时间区间、权限和状态迁移。
- 合并、拆分、重定向和依赖传播。
- 数据库事务、文件提交和错误终态。

合法 JSON 只代表“可以解析”，不代表内容已经正确。所有模型结果都必须依次经过结构校验、原文对齐和领域规则。

### 3.4 不确定性不能被文字掩盖

候选可以进入有效、不确定或拒绝状态；事实可以争议或共存；主张可以部分支持、被反驳或证据不足。系统不能为了生成一份顺畅报告而强行二选一。

### 3.5 质量上限优先，但必须实测

首版先证明质量上限，不因成本猜测提前删除可能有价值的机制；也不因为参考项目存在某机制就全部开启。昂贵路线必须可开关、可观测，并通过金标、对照和消融实验决定默认开启、条件开启、人工触发、修复、延后或删除。

### 3.6 可恢复、可诊断、可重算

长流程必须可暂停、继续、重试和局部重算。失败时说明阶段、原因、影响和下一步。原文、身份、事件边界、事实或分析方案改变后，只重算依赖链受影响部分。

### 3.7 需求决定零件

参考项目是机制、思路和实现边界的来源，不是产品功能清单。研究和采用时必须区分：原项目功能、底层机制、我们的真实需求和最终产品功能。只有需求成立且机制合适时才吸收，并按许可证独立实现或登记复用。

### 3.8 证据等级纪律

- 源码已证实：可以追到固定版本的真实数据结构、调用链、写入、读取或持久化。
- 源码推断：根据代码结构作出的合理判断，必须说明尚未实际运行。
- 实际运行已验证：只有亲自执行并观察到行为时才能使用；仓库里存在测试文件不等于已经执行。
- 产品判断：根据本项目需求作出的采用、改造、延后或舍弃决定。
- 未知：缺少固定版本、真实服务、样本或运行证据，必须标明下一步验证。

README、论文、注释、营销文案和函数名只作为线索。函数存在不等于真实主链使用；提示词中的“必须”不等于程序硬规则；拥有 ID 字段不等于稳定身份；拥有时间字段不等于检索已经时点安全。

## 4. 项目、输入和来源规则

### 4.1 项目与版本

一个项目对应一部作品，可以包含多个明确的 `SourceVersion`（来源版本）。不同修订版、译本或章节切分不得静默混合。正文改变必须生成新版本，并建立旧版到新版的章节和证据映射。

### 4.2 导入流程

```text
创建小说项目
  ↓
选择 TXT / Markdown / DOCX / EPUB
  ↓
解析编码、标题、卷章和正文
  ↓
检查顺序、空章、重复章、异常短章、超长章和乱码
  ↓
用户查看章节树和正文抽样
  ├─ 有问题：修正标题、顺序、边界或重新导入
  └─ 无问题：确认来源版本
  ↓
建立稳定原文坐标和导入制品
```

### 4.3 导入问题和处理

| 问题 | 级别 | 处理 |
|---|---|---|
| 哈希相同的重复章节 | 阻塞 | 删除、只保留一个或明确标为重复版本 |
| 章节顺序冲突 | 阻塞 | 用户确认顺序 |
| 异常编码或替换字符 | 阻塞 | 重新选择编码或导入 |
| 空章、极短章 | 警告 | 可继续，但进入问题中心 |
| 超长章 | 警告 | 分块并保留完整章节来源 |
| 标题识别不确定 | 待确认 | 用户修正，不允许标题被当正文 |
| 自动场景边界不确定 | 待确认 | 场景仅为可重建派生边界 |

### 4.4 来源与证据坐标

- 字符偏移是原文定位主权威。
- 章节序号和段落序号用于结构与用户阅读。
- 令牌偏移是模型运行派生坐标，必须记录分词器版本。
- 跨段或跨章证据用 `EvidenceSet`（证据集合）组合，不创建虚假连续范围。
- 模型只返回引用文字时，定位器必须枚举全部精确匹配；零匹配拒绝，多匹配进入不确定状态，不能选择“最近一个”。
- 正式证据至少保存来源版本、章节、起止字符、文本快照和上下文哈希。
- 界面可以显示较宽上下文，但正式引用保持最小充分范围。

### 4.5 分块规则

分块是模型输入方式，不是原文结构。系统可以使用段落窗口、场景候选窗口、章节窗口和剧情阶段窗口；每种窗口保留重叠、来源和去重信息。跨块候选按来源坐标去重，不能因分块产生多个正式对象。

## 5. 用户流程和页面

### 5.1 首次拆解流程

```text
导入并确认整本小说
  ↓
人物识别与归一
  ├─ 查看人物、别名、证据和争议
  └─ 确认或标记问题
  ↓
剧情阶段与事件识别
  ├─ 查看跨章事件、参与者、时间和证据
  └─ 确认或标记问题
  ↓
事实、世界设定、人物状态和角色认知
  ↓
伏笔、冲突、节奏和场景功能
  ↓
进入统一工作台持续浏览、搜索、关联跳转和回查原文
```

每个阶段默认停在确认页。用户确认后才继续；标记问题时优先判断问题来自原文解析、程序规则、提示词、模型能力还是来源证据，不把人工修正当作掩盖系统问题的默认办法。

### 5.2 PC 端信息架构

| 页面 | 用户目标 | 主要内容 | 关键异常状态 |
|---|---|---|---|
| 项目首页 | 创建、进入和继续项目 | 项目列表、来源版本、最近进度 | 文件丢失、项目损坏、任务中断 |
| 导入与章节确认 | 得到可信章节结构 | 文件导入、卷章树、正文预览、问题提示 | 编码错误、标题误识别、空章、重复章 |
| 阶段确认页 | 抽查并决定是否继续 | 总览、证据、问题标记、确认 | 结果为空、部分批次失败、证据失效 |
| 拆解工作台 | 浏览整本拆解结果 | 总览、人物、剧情、事件、时间线、事实状态、设定、伏笔、冲突、节奏 | 尚未生成、结果过期、上游修改 |
| 原文与证据 | 理解结论来源 | 章节树、正文、证据高亮、上下文 | 版本变化、位置失效 |
| 人物 | 追踪人物和关系 | 别名、身份、出现、关系、状态、人物弧 | 误合并、待拆分、身份争议 |
| 事件与时间线 | 追踪发生了什么 | 提及、正式事件、参与者、时间、因果和子事件 | 边界争议、共指不确定 |
| 事实与状态 | 查询某章成立的事实 | 事实版本、冲突、状态、角色认知 | 事实争议、时间不明 |
| 分析版块 | 浏览带证据的高层分析 | 剧情、伏笔、冲突、节奏、场景功能 | 证据不足、反证未处理 |
| 问题中心 | 集中处理问题 | 类型、影响对象、建议操作、重算范围 | 依赖冲突、修订影响下游 |
| 运行记录 | 了解长任务 | 阶段、批次、耗时、费用、失败、继续和停止 | 网络、限流、模型输出、恢复 |
| 设置中心 | 配置在线模型 | 模型服务、真实测试、分析方案、数据隐私 | 密钥、权限、参数、结构化输出失败 |

工作台内部固定导航为：总览、原文、人物、剧情、事件、时间线、事实与状态、世界设定、伏笔、冲突、节奏、问题。未完成模块显示“尚未分析”或“本版本尚未开放”，不能用模拟内容冒充结果。

### 5.3 人工参与边界

普通高置信结果可抽样检查。用户重点处理：来源结构错误、人物误合并、事件边界争议、事实冲突、低置信分析和受修订影响的结果。合并、拆分和关键事实裁决必须可撤销并保留历史。

## 6. 输出体系

### 6.1 三层输出

| 输出层 | 内容 | 用途 |
|---|---|---|
| 机器权威层 | 来源、证据、正式人物、事件、事实版本、角色认知、已验证主张 | 查询、重算、审计和结构化导出 |
| 用户分析层 | 时间线、人物、关系、剧情阶段、伏笔、冲突、节奏和世界规则 | 日常浏览和核验 |
| 运行审计层 | 问题、任务、版本、模型调用、令牌、费用、缓存和恢复 | 诊断、质量评测和降本决策 |

### 6.2 核心分析版块

| 版块 | 最低内容 |
|---|---|
| 全局剧情 | 章节范围、阶段目标、转折、未解决问题和证据 |
| 人物与关系 | 身份、别名、首次/最后出现、状态和关系变化、人物弧主张 |
| 事件时间线 | 原文提及、正式事件、时间、地点、参与者、时序、因果和子事件 |
| 事实与世界状态 | 世界规则、归属、能力、地点、阵营、有效区间和指定章节状态 |
| 伏笔与承诺 | 提出、强化、变形、部分兑现、兑现、失效和未回收 |
| 冲突与赌注 | 外部、内部、关系和规则冲突；升级、反转、解决和赌注变化 |
| 节奏与场景功能 | 铺垫、推进、揭示、反转、恢复、高潮和密度异常候选 |

钩子、名场面、爽点、情绪峰值、主题、母题和文风属于后续或实验分析，必须标明主观性，不能阻塞核心版本。

### 6.3 分析主张最低结构

每项重要分析至少包含：主张类型、结论、涉及对象、章节范围、支持证据、反面证据、验证状态、置信度、不确定原因和用户可读解释。

### 6.4 导出

工作台是主体。后续可提供 JSON 结构化包、CSV 指标、Markdown/HTML 报告和 DOCX；导出不得默认包含整部原文，只包含用户选择的引用和定位。

## 7. 总体系统形态

### 7.1 架构选择

首版采用本地优先的模块化单体：PC 浏览器界面、API 进程和本机 Worker（后台执行进程）共享同一领域模型、SQLite 权威数据库和文件制品目录。

```text
PC 浏览器界面
  ↕ HTTP + 进度流
应用 API
  ├─ 用户命令与参数校验
  ├─ 只读查询与页面投影
  └─ 任务登记
        ↕
本机 Worker
  ├─ 长时间模型调用
  ├─ 分批分析与局部重算
  └─ 制品提交与恢复
        ↕
SQLite + 不可变文件制品 + 可重建索引 + 在线模型服务
```

API 不在浏览器请求内同步运行整本分析；浏览器断开不能中断后台任务。首版不引入微服务、Redis、RabbitMQ、云数据库或图数据库。

### 7.2 组件职责

| 组件 | 负责 | 不负责 |
|---|---|---|
| PC 界面 | 项目、证据、问题、工作台和运行状态 | 直接修改数据库、决定稳定身份 |
| API | 参数校验、命令登记、查询投影和本地安全边界 | 同步执行长分析 |
| Worker | 领取任务、调用模型、提交制品、更新进度 | 绕过领域规则直接写权威对象 |
| 领域服务 | 身份、边界、事实、状态、检索、主张和依赖规则 | 依赖具体页面或模型供应商 |
| 模型网关 | 请求、超时、重试、限流、能力和用量 | 决定业务状态和权威写入 |
| 存储层 | 事务、版本、来源、索引和文件持久化 | 把派生摘要提升为事实 |

### 7.3 技术边界

- Python 后端，FastAPI 接口，Pydantic/JSON Schema 数据校验，SQLAlchemy/Alembic 数据库迁移。
- React + TypeScript + Vite 的 PC 端界面，只依赖 API 的用户投影，不保存领域权威。
- SQLite WAL 为首版权威事务库；全文检索、向量和图是可重建派生索引。
- 大型 JSON、报告和批次结果按内容哈希保存为文件，数据库登记身份、路径、状态和来源链。
- 在线模型和向量模型通过可替换接口接入，不绑定单一供应商。

### 7.4 Windows 启动与进程生命周期

- 最终用户只使用项目根目录的 `启动AI小说拆解工作台.bat`，不需要分别启动前端、API 和 Worker。
- 启动入口保留一个可见窗口，依次检查数据库、启动 API、启动 Worker、启动页面并自动打开浏览器；窗口显示当前状态、页面地址、日志位置和关闭方式。
- 页面固定使用 `http://127.0.0.1:15173`，API 固定使用 `http://127.0.0.1:18000`，避免与 SillyTavern 默认使用的 `8000` 端口冲突。
- 浏览器断开、刷新或关闭标签页不能中断数小时的分析任务；可见启动窗口才是本机系统的生命周期开关。
- 启动器只允许一个正式实例运行。API、Worker、前端及派生进程必须加入同一 Windows 作业组；正常关闭、按 `Ctrl+C`、启动失败或启动器进程意外消失时，作业组统一回收子进程。
- `dev.ps1` 和产品预览脚本只用于开发者调试，不能作为用户入口，也不能留下与正式数据库共用的旧 Worker。
- 每次运行使用独立日志目录；启动失败必须在窗口中说明失败步骤和日志位置，不允许安静失败。

## 8. 数据所有权和分层

### 8.1 权威顺序

```text
原始小说与来源版本
  ↓
章节、段落和原文证据
  ↓
人物 / 事件 / 事实候选
  ↓ 结构校验 + 原文对齐 + 领域规则
正式人物 / 事件提及 / 正式事件 / 事实版本
  ↓
人物状态、角色认知、关系图、剧情阶段和检索视图
  ↓
带支持与反证的分析主张
  ↓
伏笔、冲突、节奏等分析制品和工作台页面
```

上游修改必须标记受影响下游。来源、正式对象、派生视图、分析主张和自然语言报告分开保存。

### 8.2 对象所有权

| 对象域 | 核心对象 | 权威边界 |
|---|---|---|
| 来源 | Project、SourceVersion、Chapter、SourceUnit、EvidenceSpan | 原文与坐标最高权威 |
| 候选 | Entity/Event/Fact/Claim Candidate | 模型和规则提议，不能直接晋升 |
| 身份 | EntityMention、CanonicalEntity、EventMention、CanonicalEvent | 人物与事件稳定身份 |
| 事实 | Fact、FactVersion、FactEvidence、FactConflict | 事实命题、版本、区间和冲突 |
| 状态 | StatePatch、State-at-T | 从事实确定性重放的派生状态 |
| 认知 | ActorKnowledge | 某角色知道、相信、怀疑或隐瞒什么 |
| 检索 | TaskProfile、RetrievalCandidate、RetrievalRun | 任务材料选择和来源记录 |
| 主张 | AnalysisClaim、ClaimEvidence、ClaimJudgment | 已验证分析结论 |
| 叙事制品 | Promise、Decision、Arc、Report | 只消费验证结果的派生制品 |
| 运行 | Run、Task、TaskAttempt、Artifact、Blob、Issue、Metric、ModelCall | 任务、恢复、来源链和观测 |

### 8.3 单向晋升

```text
模型原始输出
→ 可解析候选
→ 数据结构校验
→ 原文位置校验
→ 领域规则校验
→ 有效 / 不确定 / 拒绝
→ 仅有效结果写入正式对象；不确定结果进入问题中心
```

禁止从“看起来合理”直接写正式对象。

## 9. 稳定身份、状态和审计字段

### 9.1 身份原则

- 来源内定位对象使用由版本和坐标计算的确定性身份。
- 跨版本人物、事件、事实和主张使用程序生成的持久身份。
- 自然语言名字、标题或模型返回 ID 不能作为主键。
- 合并不删除旧身份；旧身份重定向到保留对象并保留来源链。
- 拆分创建新身份，记录派生关系，不能静默复用旧身份表达多个对象。

### 9.2 身份类型

| 对象 | 身份依据 | 变化规则 |
|---|---|---|
| 来源版本 | 项目 + 标准化全文哈希 | 正文改变则新版本 |
| 章节 | 来源版本 + 顺序 + 标题/正文特征 | 新版本重建并映射 |
| 原文证据 | 来源版本 + 起止字符 + 文本哈希 | 文本或边界改变则新身份 |
| 人物/事件提及 | 原文证据 + 局部边界 | 原文或边界改变则新身份 |
| 正式人物/事件 | 程序生成持久身份 | 合并/拆分记录来源链 |
| 事实与事实版本 | 命题族身份 + 版本身份 | 追加版本，不覆盖历史 |
| 分析主张 | 程序生成身份 + 版本 | 修复产生新版本或派生主张 |
| 制品 | 类型 + 输入哈希 + 配置/代码/提示词版本 | 内容寻址且不可变 |

### 9.3 统一审计字段

正式对象至少记录项目、来源版本、状态、创建和更新时间、生成阶段、输入制品、证据、数据结构版本、代码版本、提示词版本、模型方案、置信度、不确定原因、替代对象和来源链。

### 9.4 核心状态

| 对象 | 状态 |
|---|---|
| 候选 | PROPOSED / VALID / UNCERTAIN / REJECTED |
| 正式人物/事件 | ACTIVE / MERGED / SPLIT / RETIRED |
| 事实版本 | ACTIVE / SUPERSEDED / INVALIDATED / DISPUTED / COEXISTING |
| 分析主张 | PROPOSED / SUPPORTED / PARTIAL / DISPUTED / CONTRADICTED / INSUFFICIENT / DROPPED |
| 制品 | WRITING / READY / FAILED / DIRTY / STALE / SUPERSEDED |

## 10. 人物身份归一

### 10.1 候选来源

人物候选可来自名称/别名、上下文语义、关系与角色、时间和共同出现约束以及用户词典。候选发现追求召回，正式合并优先保证精确率。

### 10.2 归一链

```text
人物提及候选
→ 名称、属性、关系和时间阻塞
→ 候选对及证据
→ 同一 / 不同 / 不确定
→ 聚类提议
→ 全聚类矛盾检查
→ 正式人物或待确认问题
```

### 10.3 硬性边界

- A 像 B、B 像 C 不代表 A 必然像 C，禁止无保护传递合并。
- 同场同时出现的独立角色、互斥属性和用户锁定“不同”是硬负例。
- 用户可锁定同一或不同，后续模型必须尊重。
- 合并、拆分和撤销会把相关事件、事实、状态、主张和报告标为过期并局部重算。

## 11. 事件发现、边界和跨章事件

### 11.1 三层身份

- `EventCandidate`：某条路线提出的候选。
- `EventMention`：原文中的一次具体提及，拥有稳定边界。
- `CanonicalEvent`：多个提及归并后的同一叙事事件。

三者不能混用。

### 11.2 候选路线

首版至少组合模型结构化提取和规则/状态变化两类路线；以后可增加触发词、事件类型查询、参数聚集和时间溢出路线。每条候选保留路线、模型、原始响应、文本、边界、参与者、时间、地点、置信度和证据。

### 11.3 原文对齐

| 输入 | 处理 |
|---|---|
| 字符位置和文字一致 | 接受对齐 |
| 只有令牌位置 | 用相同分词器投影到字符并记录版本 |
| 引用文字唯一匹配 | 补齐字符范围 |
| 多处精确匹配 | 保留全部位置候选并标记不确定 |
| 只有模糊改写 | 不直接落地，进入二次定位或人工确认 |
| 跨句描述 | 核心触发范围 + 多条支持证据，不制造虚假长范围 |

### 11.4 候选并集和边界裁决

多路线候选进入统一格式后，按同一窗口、范围重叠、包含、文本、类型、参与者和时间聚类。边界裁决选择最小充分触发范围；路线不一致或边界差异过大时进入不确定状态，不能取平均或选择最近同词。

### 11.5 正式事件和关系

正式事件归并综合事件类型、参与者、故事时间、地点、对象、叙事关系和文本语义。支持同一事件、前后/重叠、因果/条件、子事件以及报告、回忆、想象等叙事关系。想象、传闻和回忆不能自动当作当前真实发生。

## 12. 事实版本、状态和角色认知

### 12.1 事实与版本

`Fact` 表示稳定命题族，`FactVersion` 表示命题在某个有效区间的具体版本和证据。至少保存主语、谓词、对象/数值、观察章节、有效起止、可选故事时间、陈述方式、认知范围、置信度和状态。

### 12.2 更新规则

| 场景 | 表示 |
|---|---|
| 新事实替代旧事实 | 旧版本 SUPERSEDED，新版本从新章节生效 |
| 事实失效后再次成立 | 追加非连续版本，不覆盖旧区间 |
| 多个值同时成立 | 各自 COEXISTING |
| 来源互相矛盾 | DISPUTED，保留支持与反证 |
| 角色错误相信 | 写入角色认知，不写成读者事实 |
| 假设、计划、愿望 | 写入主张、承诺或角色认知，不当作已发生事实 |

禁止使用一个“当前值”覆盖历史，禁止把摘要中的事实直接写回事实库，禁止让模型选择任意数据库身份。

### 12.3 状态投影

事实库负责权威命题和版本；状态投影器按指定章节或故事时间，对经过校验的状态变更做确定性重放。状态快照是缓存，可从事实和变更重建，不是新的事实权威。

### 12.4 角色认知

角色认知与读者真相分开，至少支持：已知、相信、怀疑、未知、隐瞒和撤回旧信念。以角色视角查询时，在指定时间的客观状态上应用该角色可见性和信念，不泄露未来知识。

## 13. 任务相关检索和上下文选择

### 13.1 任务决定检索路线

不同任务不固定串行执行所有检索。`TaskProfile`（任务配置）决定范围、时间、人物、启用路线、每路配额、重排方式、是否必须反证、是否允许高成本全局路线以及模型方案。

### 13.2 路线

- 全文/词法：精确名字、原句、稀有词和规则。
- 向量：语义相似和改写。
- 人物/事件/事实：围绕正式对象扩展。
- 时间：指定章节、区间和前后变化。
- 局部图：参与者、物品、地点和相邻关系。
- 图扩散：从多个种子有界扩展。
- 社区/全局：全书结构和主题导航。
- DRIFT：复杂全局问题逐步下钻。

### 13.3 统一检索链

```text
任务意图
→ 路线激活
→ 各路线构造查询
→ 候选并集和统一格式
→ 分数归一/融合
→ 一次主重排
→ 路线配额和多样性
→ 回填原文来源
→ 最终上下文落点检查
```

不能把多个项目的融合、重排和图查询机械串联。全局摘要和社区报告只负责导航；最终上下文必须包含原文证据或可回源正式对象，并记录每条材料来自哪条路线。

### 13.4 条件触发，不做世界书

可吸收关键词、次级条件、优先级、预算和有界递归，用于选择相关人物、设定和历史证据。产品不创建世界书页面，不继承其字段、编辑器或聊天注入方式。

### 13.5 上下文预算

每次请求先预留任务约束、输出结构、当前原文、必须保持一致的正式对象和回复空间。剩余预算再按任务选择历史证据、世界设定、上游结构和示例。超出预算时记录省略内容和原因，不能静默截断原文。

故事结构和深层分析在长篇输入中先生成约 20～40 个连续章节范围导航；短篇则按实际章节数保留。每个范围记录起止章节、代表标题、代表事件、参与对象、事件与证据引用以及省略事件数量，用来证明全书哪些范围已经被看见。该导航明确标记为派生结果，只负责防止中间章节在全局请求中静默消失，不能代替正式事件或原文证据。分析方案已支持记录服务明确提供的模型上下文长度；有可靠数值时从上下文窗口扣除最大输出和安全预留，没有可靠数值时使用保守自动预算，并在运行诊断中记录预算来源。

## 14. 分析主张、验证和专项分析

### 14.1 主张优先

专项分析器先输出最小可判定的结构化主张，不先写长文再反向猜证据。主张包含类型、对象、章节范围、命题、限定条件、所需证据类型、初始证据和版本。

### 14.2 验证链

```text
结构化主张
→ 检索支持证据
→ 检索反面或例外证据
→ 逐条判定
→ 冲突感知聚合
→ 支持 / 部分支持 / 争议 / 反驳 / 证据不足
→ 拆分 / 缩小范围 / 增加限定 / 补检索 / 删除
→ 已验证主张账本
```

逐条证据可标为支持、部分支持、反面、无关或模糊。禁止“只要一条证据支持，就把整条主张判为支持”；强支持与强反证同时存在时应进入争议或缩小主张成立范围。

### 14.3 六类核心分析器

| 分析器 | 主要输入 | 输出 |
|---|---|---|
| 剧情结构 | 事件、事实、章节功能、全局检索 | 阶段、转折、目标、障碍和剧情弧 |
| 人物与关系 | 人物、指定章节状态、角色认知、事件 | 人物弧、动机和关系变化 |
| 伏笔与承诺 | 事件、主张、词法和时间检索 | 提出、强化、变形、兑现和失效 |
| 冲突与赌注 | 事件、目标、关系和世界规则 | 类型、升级、反转、解决和赌注变化 |
| 节奏与场景 | 章节/场景指标、事件、对话和动作 | 场景功能、密度、转场和节奏异常 |
| 世界规则 | 事实版本、例外和状态 | 规则、限制、代价、例外和违反 |

### 14.4 报告编译和反馈隔离

报告只消费正式对象和已验证主张，自动插入来源，检查矛盾和缺失引用。用户修改报告文字时，应转化为修订主张、处理问题或增加人工注释，不能把报告文本写回事实库。

## 15. 后台任务可靠性

P0 可靠性已经实现并通过自动测试和 Windows 强制中断恢复验证。以下契约继续约束所有后续长任务。

### 15.1 不变量

1. 一个任务最多只有一个当前有效尝试。
2. 领取任务是数据库原子事件；未领取成功的 Worker 不得调用模型。
3. 尝试次数只在成功领取时增加。
4. 每次领取生成新的尝试身份、租约代次和不可复用令牌。
5. 心跳、成功、失败、制品预留和发布都验证完整租约身份。
6. 旧 Worker 的迟到响应不能修改任务或制品终态。
7. 达到最大次数的任务必须失败或取消，不能永久运行。
8. 成功、失败和取消是普通执行路径不能反向离开的终态。
9. 同一任务的一种最终结果只有一个制品身份，内容去重不能破坏来源链。
10. 数据库和文件系统的中间状态必须可扫描、判断和恢复。
11. 模型调用期间不持有数据库写事务。
12. 时间、令牌、退避和故障点可注入，测试不依赖真实长时间等待。

### 15.2 任务和尝试状态

```text
PENDING → RUNNING → SUCCEEDED
                  → RETRY_WAIT → RUNNING
                  → FAILED
                  → CANCEL_REQUESTED → CANCELLED
PENDING / RETRY_WAIT → CANCELLED
```

尝试状态包括运行、成功、可重试失败、永久失败、租约过期、取消和旧尝试。`max_attempts=3` 表示最多真实执行三次，包括第一次。

### 15.3 原子领取和租约隔离

SQLite 使用短事务和条件更新领取任务；领取事务内禁止模型、文件和网络操作。Worker 领取后关闭会话，使用独立心跳续租。所有提交携带任务、尝试、令牌、代次和 Worker 身份；不匹配返回租约丢失、旧尝试、已请求取消或任务已终止。

### 15.4 重试和取消

过期扫描器处理租约过期：已请求取消则取消；次数耗尽则失败；否则进入等待重试。超时、限流和临时不可用可以重试；认证、错误参数和不支持任务永久失败。取消等待任务立即生效；取消运行任务进入“正在取消”，随后提交被租约条件拒绝。重复取消幂等。

### 15.5 Provider（模型服务）失败约定

模型服务通过注册表注入，不由任务内部直接创建。稳定错误至少包括：超时、限流、临时不可用、非法输出、认证失败、错误请求和不支持任务。错误对象必须携带稳定错误码、是否可重试和建议等待时间。

## 16. 制品、崩溃恢复和局部重算

### 16.1 制品和 Blob 分离

`Artifact`（制品）表示某个任务结果及来源链；`Blob`（内容文件）按哈希存放不可变内容。不同任务内容相同可以共用 Blob，但必须拥有各自的 Artifact。

### 16.2 提交流程

```text
模型返回并通过校验
→ 写尝试专属临时文件并计算哈希
→ 在租约条件下预留 WRITING 制品
→ 原子提升为内容寻址 Blob
→ 再次验证租约、代次和哈希
→ 同一数据库事务完成：制品 READY + 任务成功 + 尝试成功
```

临时文件、WRITING 记录、孤儿 Blob、缺失文件和哈希不匹配都必须能由恢复器扫描。重复恢复安全；旧尝试结果标为 STALE，不能覆盖当前结果。

### 16.3 依赖和 DIRTY（过期）传播

| 上游变化 | 直接受影响 | 后续传播 |
|---|---|---|
| 原文/证据 | 人物、事件和事实候选 | 正式对象、索引、主张、报告 |
| 人物合并/拆分 | 事件、事实和角色认知 | 状态、关系图、分析和报告 |
| 事件边界/归并 | 事件、事实和伏笔 | 时间线、剧情/冲突分析、报告 |
| 事实版本 | 状态和世界/人物主张 | 报告和导出 |
| 提示词/模型/方案 | 该阶段制品 | 只传播到依赖该制品的下游 |
| 报告模板 | 报告 | 不重算人物、事件、事实和主张 |

源文本修改产生新来源版本，通过哈希和映射找到受影响证据，沿依赖图标记过期并重算最小子图。旧制品保留版本和差异，不能直接覆盖。

## 17. 在线模型、分析方案和提示词

### 17.1 设置中心只有两层

- 模型服务：名称、类型、接口地址、访问密钥、连接状态和模型目录。
- 分析方案：某类任务使用哪个服务、模型、参数、提示词和失败策略。

一个服务可以供多个任务使用；人物抽取、事件分析、事实核对和全书归纳不应共用一套固定参数。

### 17.2 连接成功的层级

界面必须区分配置已保存、地址和密钥可访问、模型目录读取成功、所选模型有权限、所选模型支持当前任务的结构化输出和参数。“读取到 20 个模型”不能显示成“已经可以拆书”。

### 17.3 参数原则

- 没有任务实验依据时使用“自动”，不发送温度、采样或推理字段。
- 模型或服务不支持的参数不显示、不发送。
- 输出长度按连续整数设置，不使用无依据的 256 步长。
- SillyTavern 的聊天默认温度 1.0 和项目旧值 0.2 都不能直接成为小说任务默认值。
- 每次运行记录实际服务、模型、参数、价格快照和方案版本。

### 17.4 能力记录

服务能力至少记录模型目录、普通请求、严格结构化输出、普通 JSON、推理强度、最大输出字段、流式响应、上下文和输出上限。能力来自已知规则、低成本真实测试和用户手工覆盖，不能只按模型名猜。

### 17.5 提示词编排

提示词不是一个巨型文本框，而是按任务保存目标、原文、正式对象、约束、输出结构、示例和失败降级等组件。每个组件有标识、版本、顺序、适用任务和令牌数。运行记录保存最终实际请求的脱敏版本、原始响应、解析结果和修复尝试。

### 17.6 密钥和配置恢复

密钥与普通设置分开保存；前端默认只知道是否设置，不回传完整值。密钥不进入项目导出、日志和 Git。模型服务和分析方案保存前保留配置快照；项目数据、模型设置和密钥分别备份，恢复前说明覆盖范围。

### 17.7 长调用反馈

点击开始后立即显示当前阶段、批次、整体进度和预计剩余时间。停止后先显示“正在停止”，后台在安全边界取消。用户离开页面任务继续，返回后恢复真实状态。完成、部分完成、停止和失败必须有不同说明和下一步。

## 18. API、隐私和安全

### 18.1 命令与查询分离

创建项目、导入、开始分析、解决问题、合并人物和重算属于命令；项目、证据、人物时间线、状态、主张和运行指标属于只读查询。界面不能提交任意状态、数据库身份或审计字段。

### 18.2 本地与隐私

- 默认只监听 localhost。
- 小说、项目数据、数据库、制品、日志和密钥不提交 Git。
- 模型服务方案明确是否允许发送原文、最大窗口和脱敏要求。
- 日志不打印完整章节，只记录哈希、证据身份和必要短片段。
- 导出不默认包含整部原文。
- 删除项目和清理模型原始响应需要明确确认和审计。

### 18.3 安全边界

- 小说正文作为数据块，与系统指令和输出结构隔离，降低提示词注入风险。
- 所有文件路径由工作区服务解析，禁止路径穿越。
- 模型输出不能作为 SQL、Python 或模板代码执行。
- 密钥通过本机安全存储，不由普通 API 回传。
- 制品使用哈希、原子写入、结构校验和恢复对账。

## 19. 质量、金标和验收标准

### 19.1 测试语料

使用约 5 万字、50 万字和 100 万字以上样本验证。语料应包含别名、假身份、同名人物、关系变化、跨章事件、回忆/转述、事实失效和重新成立、角色误解、伏笔与普通描写并存以及章节修订。

### 19.2 金标原则

被评测的同一模型不能自动生成并直接确认答案。AI 可以提出候选，但高风险金标需人工确认并保留争议说明。至少 20% 高风险样本进行第二轮复核。

建议金标覆盖人物提及、正式人物、事件提及、正式事件、事实版本、指定章节状态、分析主张和伏笔生命周期；数量随样本规模调整，不再绑定旧 30～50 章固定数量。

### 19.3 硬性闸门

- 正式对象的证据回贴率 100%。
- 正式对象数据结构校验通过率 100%。
- 孤儿引用为 0。
- 同输入重复运行不产生重复正式对象。
- 模型超时、非法输出、进程中断和局部重算均可恢复。
- 证据错位、跨项目污染、无效身份和不可恢复的 P0 错误为 0。

### 19.4 初始质量目标

这些是首轮实验目标，不是宣传承诺；调整必须记录样本、原因和影响。

| 对象 | 初始目标 |
|---|---:|
| 人物归一精确率 | ≥ 0.95 |
| 人物归一召回率 | ≥ 0.85 |
| 事件提及发现召回率 | ≥ 0.80 |
| 事件边界宽松 F1 | ≥ 0.85 |
| 事件边界精确 F1 | ≥ 0.70 |
| 跨章正式事件归并 F1 | ≥ 0.80 |
| 事实正确率 | ≥ 0.90 |
| 事实有效区间准确率 | ≥ 0.85 |
| 指定章节状态准确率 | ≥ 0.90 |
| 主张证据充分率 | ≥ 0.90 |
| 最终核心主张无充分证据比例 | ≤ 0.05 |

专项分析人评关注事实正确、证据充分、结构洞察、不确定性和可用性。核心事实错误率不高于 5%，重要主张证据覆盖 100%。

### 19.5 错误等级

| 级别 | 示例 | 处理 |
|---|---|---|
| P0 阻断 | 证据指错、身份冲突、跨项目污染、无法恢复 | 必须修复，不能验收 |
| P1 严重 | 人物误合并、关键事件漏失/误归并、错误事实写入 | 修复并重跑受影响评测 |
| P2 重要 | 主张证据不足、伏笔过度解读、时间因果错误 | 进入问题中心，不能发布 |
| P3 一般 | 措辞、排序和次要可读性 | 后续迭代 |

### 19.6 对照和消融

至少比较普通章节摘要+全局总结、只有原文检索+结构化主张、完整质量模式。优先测试关闭多路线事件、反面证据、人物归一、图/社区、指定时点状态和二次主张修复后的质量与成本变化。

结果处理：质量显著则保留；只对特定任务有效则条件开启；质量增益小但昂贵则人工触发或延后；可通过缓存/批处理改善则修复后复测；连续两轮仍不稳定或引入严重错误则删除。

### 19.7 观测字段

每次模型调用和阶段运行记录：运行/阶段/任务/来源和制品版本、服务/模型/提示词/参数、输入输出和缓存令牌、字符和对象数、开始结束和耗时、重试/超时/错误码、费用与价格快照、金标子集、问题数和接受/修复/拒绝数量。分析方案可保存用户从服务商取得的输入/输出单价和币种；单价未知时不显示金额。开始前按当前分批、全局分析次数和最大输出计算保守上限，运行后按服务实际返回的令牌和当次价格快照计算费用；后续修改单价不得改写历史结果。

归一化指标包括每 1 万字的令牌、调用、耗时和成本；每章、事件、事实和已验证主张成本；缓存命中、局部重算节省和各路线的质量增益/增量成本。

### 20.1 关键负面边界

- 不使用只按名字精确检索、标题相等合并或随机上下文。

- 不用无保护传递闭包合并人物。
- 不把自然语言因果数组、名字/主题相等写入当稳定事件。

- 不采用“任一支持证据存在即主张成立”。
- 不把社区报告当原文证据。
- 不把物理编辑/删除冒充不可变事件历史。

## 22. 当前开发顺序和交付标准

### 22.1 P1/P2 当前收口

1. 模型自动参数、所选模型测试和能力过滤已经实现自动验证；真实服务兼容性随第一次验收实测。
2. 按 P19 完成任务级提示词版本、无敏感请求摘要、结构校验位置、调用/重试次数、令牌用量、预算内材料构成、可靠单价、分析前保守费用上限和运行后实际费用展示；超出模型安全上下文的材料已自动执行“连续章节范围 → 阶段 → 全书”的可恢复聚合，仍需补更多服务的上下文和价格元数据自动吸收以及真实长篇在线验证。
3. 完成复杂 TXT/Markdown/DOCX/EPUB 导入、卷章校正和来源版本体验。
4. 完成人物档案、保守人物归一投影、剧情阶段、事件时间线和跨章证据回查。
5. 以上能力目前是内部首条闭环，不安排用户逐步验收；待 P3/P4/P5 形成完整工作台后统一进行第一次真实体验。

### 22.2 P3 事实与状态

首版已实现事实有效章节区间、指定章节状态读取、完整事实版本历史、事实失效/再次成立/冲突共存、状态变化、人物认知、世界规则和证据回查。下一步补跨对象状态重放、认知传播校验和事实问题的细粒度重算；成功标准是用户能选择章节，看到当时成立的事实、人物状态、角色知道什么以及全部来源。

### 22.3 P4 核心分析

首版已实现结构化主张、支持/反证/证据不足、伏笔、冲突、节奏和场景功能；全书章节范围导航及“范围 → 阶段 → 全书”层级产物已经防止长篇中间部分在全局请求中静默消失，每层保留向下来源边且只作为导航。下一步补逐条复核、主张修复、跨制品一致性和真实长篇质量评测；成功标准是每项核心结论可回原文，强反证未处理时不能发布为已支持。

### 22.4 P5 统一工作台

首版已完成正式 PC 导航、全局搜索、整章原文并排查看、原文证据回查、指定章节状态、故事结构确认门、完整深层拆解确认门、问题中心、问题驱动的深层重新分析、历史版本选择和版本差异摘要；页面不再把作品信息或内部单元序号冒充小说章节，也不再在常见桌面缩放下横向裁切。下一步补关联跳转、影响范围、更加细粒度的局部重算和版本逐项质量审计；成功标准是用户不接触内部任务编号，也能理解当前阶段、错误和下一步。

### 22.5 P6 长篇验收

使用多篇幅样本验证质量、时间、费用、恢复、等待体验和结果可读性；根据消融数据设计后续平衡/经济模式，再补完整导出和正式入口。

### 22.6 开发准入

每个用户功能编码前必须写清用户目标、页面入口、成功/空白/加载/失败/恢复状态、权威对象、参考机制、明确不采用的做法、交付物和验证。模拟数据和内部编号只能出现在调试入口。

## 23. 用户验收边界

用户只验收产品功能和使用效果，不验收分支、PR、数据库迁移、任务租约、制品编号和自动测试。

正式验收节点：

1. 能导入真实整本小说、正确显示章节，并得到真实人物和事件结果。
2. 人物、剧情、事件、事实、伏笔、冲突等核心版块完整可浏览，并能回查原文。
3. 整本长篇从导入连续运行到完整工作台。
